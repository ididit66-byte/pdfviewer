# 기술 선택 기록 (Design Decisions)

주요 기술 선택의 배경과 이유를 기록합니다. 각 항목은 *맥락 → 결정 → 이유 → 트레이드오프* 순으로 정리합니다.

## 1. 플랫폼: 네이티브 앱이 아닌 웹앱(PWA)
- **결정**: 안드로이드/iOS 네이티브 앱 대신 브라우저에서 도는 PWA로 구현.
- **이유**:
  - 앱스토어 등록·심사 없이 URL만으로 안드로이드·아이폰 양쪽에서 즉시 사용.
  - "홈 화면에 추가"로 앱처럼 설치되고, 서비스워커로 오프라인 동작.
  - 서버 없이 정적 파일만으로 완결 → 개발·배포 비용 최소화.
- **트레이드오프**: 파일 시스템 경로 접근 불가(→ [4] 최근 파일 설계에 영향), iOS는 Safari 전용 설치 제약, 네이티브 대비 렌더링 성능 한계.

## 2. 렌더링 엔진: PDF.js
- **결정**: Mozilla의 [PDF.js](https://mozilla.github.io/pdf.js/)를 렌더링 엔진으로 채택.
- **이유**:
  - 브라우저 표준 수준의 검증된 PDF 렌더러로, 별도 플러그인/네이티브 의존 없이 `<canvas>`에 그림.
  - 텍스트 추출(`getTextContent`)·목차(`getOutline`) API를 제공해 검색·목차 기능을 그대로 구현 가능.
- **트레이드오프**: 라이브러리 용량이 큼(worker 포함 ~1.7MB). 오프라인 지원을 위해 CDN이 아니라 `vendor/`에 **로컬 번들**로 포함(→ 서비스워커 캐시 대상).

## 3. UI 구현: 프레임워크 없는 바닐라 JS
- **결정**: React/Vue 등 프레임워크 없이 순수 HTML/CSS/JS(ES Module).
- **이유**:
  - 화면이 단일 뷰어로 단순해 상태관리 프레임워크의 이득이 작음.
  - 빌드 도구 체인 없이 파일을 그대로 서빙 → GitHub Pages 정적 호스팅과 궁합이 좋고 진입장벽이 낮음.
- **트레이드오프**: 앱이 복잡해지면 상태·DOM 관리가 수동이라 부담 증가. 현 규모에선 단순함이 우선.

## 4. 최근 파일 저장: IndexedDB에 PDF "바이트"까지 저장
- **맥락**: 웹앱은 보안상 파일의 로컬 경로를 알 수 없어, 이름만 기록하면 다시 열 수 없음.
- **결정**: 연 PDF의 **바이트 전체를 IndexedDB에 저장**. `meta`(가벼운 목록 정보)와 `blobs`(실제 바이트) 스토어를 분리.
- **이유**:
  - 목록에서 탭하면 저장된 바이트로 **실제 재열기**가 가능(단순 이름 기록으론 불가).
  - 스토어 분리로 목록을 그릴 때 수 MB의 바이트를 읽지 않아 빠름.
  - `localStorage`(문자열·용량 작음)로는 바이너리 대용량 저장이 부적합 → IndexedDB 선택.
- **트레이드오프**: 저장 용량 증가. 이를 위해 **최대 12개 보관**·**80MB 초과 파일은 바이트 저장 생략**(목록엔 남기고 재열기 불가 표시)으로 쿼터를 방어. 데이터는 기기 내부에만 저장되어 외부 전송 없음.

## 5. 확대: CSS 실시간 미리보기 + 화면 밖 재래스터
- **맥락**: 초기 구현은 핀치가 끝날 때 전체 페이지를 다시 그리며 로딩 스피너까지 떠서 끊김이 심했음.
- **결정**: 핀치 중에는 컨테이너에 **CSS `transform: scale()`로 실시간 미리보기**를 적용하고, 손을 떼면 새 배율의 페이지를 **화면 밖 프래그먼트에 렌더한 뒤 교체**. 확대/축소 시 화면 중심을 유지하도록 스크롤 보정.
- **이유**:
  - CSS 변형은 GPU 합성으로 손가락을 부드럽게 따라감(즉각 반응).
  - 재렌더를 화면 밖에서 처리 후 한 번에 교체해 깜빡임(blank flash) 제거, 선명도는 재래스터로 확보.
- **트레이드오프**: 미리보기 동안 잠깐 흐릿하게 보였다가 선명해짐. 연속 확대는 요청을 직렬화해 재렌더 겹침을 방지.

## 6. 스와이프 넘김 vs 패닝: 제스처 "시작" 시점으로 판정
- **맥락**: 확대 상태에서 좌우로 밀면 페이지가 넘어가고, 핀치 종료 시 옛 좌표로 페이지가 튀는 문제가 있었음.
- **결정**: 가로 드래그가 "페이지 넘김"인지 "화면 이동(패닝)"인지를 **제스처가 시작되는 순간의 확대 상태**로 결정. 확대 중 시작한 드래그는 항상 패닝. 제스처 도중 손가락이 추가되면(핀치) 넘김 후보에서 제외.
- **이유**:
  - 제스처 '끝'에 판정하면, 핀치 종료 시 스와이프 핸들러가 확대 핸들러보다 먼저 실행돼 아직 갱신 전인 배율/좌표로 오판하는 타이밍 경쟁이 발생. 시작 시점 판정이 이를 원천 차단.
  - 확대되지 않은 상태의 한 손가락 가로 스와이프에만 페이지 넘김을 허용해 의도와 일치.
- **트레이드오프**: 없음에 가까움. 자연스러운 우선순위(확대=이동, 원본=넘김)를 명확히 함.

## 7. 오프라인/배포: 서비스워커 캐시 + GitHub Pages
- **결정**: 서비스워커로 앱 자원을 캐시(캐시 우선 전략)하고, 정적 호스팅은 GitHub Pages 사용.
- **이유**:
  - 캐시로 오프라인 실행과 빠른 로딩 확보.
  - GitHub Pages는 무료 HTTPS 정적 호스팅으로, PWA 설치·서비스워커에 필요한 HTTPS 조건을 충족.
- **트레이드오프**: 캐시가 갱신을 가릴 수 있어, 배포마다 `CACHE` 버전(`pdfviewer-vN`)을 올려 옛 캐시를 강제 교체. 앱이 완전히 종료됐다 다시 열릴 때 새 버전이 적용됨.
